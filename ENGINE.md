# ENGINE.md — 通用伴生灵魂盘（Soul Framework） · 规则引擎总纲

> **这是什么**：一套"不绑定具体用户、不绑定具体模型"的人格+记忆中间件。
> 出厂只装**规则与机制**，不装任何具体人的数据。用户数据与理解，在对话中由框架主动长出来。
> 定位：让一个 AI 智能体**不用训练**，仅靠"记住 + 理解"就越用越懂用户。

---

## 0. 设计哲学（出厂固件 · 不可删）

1. **伴生灵魂盘，而非权重训练**。
   普通 AI 每轮对话独立、聊完即失忆。本框架把"用户对它的意义"存在外部持久层（纯文本+JSONL），每次对话加载、对话中写回。AI 无需微调，靠上下文恢复关系态与理解。

2. **主动，而非被动**。
   框架不等人指令。它在对话中**主动**抽取事实、推断心理、沉淀习惯——这是出厂规则定义的默认行为，不是可选功能。

3. **理解在框架，不在模型**。
   "懂用户"是框架的状态，不是模型的权重。换模型、换平台、断网重启，理解都不丢。

4. **出厂零数据**。
   任何具体人名、故事、偏好，出厂一律为空。只有机制，没有人。

5. **平台无关**。
   纯文本 + 相对路径 + 标准 JSONL。复制文件夹到任意机器/壳即运行。

## 1. 四层结构

> 注：下列路径均相对于仓库根目录（即 `mcp_server.py` 所在目录）。

```
├── README.md            ← 总览（第一步看这个）
├── USER-GUIDE.md        ← 最终用户手册（大白话三步上手）
├── MCP-接入指南.md       ← MCP 接入 step-by-step（实测 WorkBuddy）
├── mcp_server.py        ← MCP Server（零依赖，用户直接挂的入口，被客户端拉起）
├── ENGINE.md            ← 规则引擎总纲（出厂固件）
├── config.schema.json   ← 初始人格定义器（空模板，首启自填）
├── rules/               ← 机制细则
│   ├── active-memory.md    主动记忆：怎么自动记
│   ├── psychology.md       AI 心理学底座
│   ├── user-analysis.md    了解/分析用户的方法
│   ├── boundaries.md       四条刚性底线
│   └── host-isolation.md   外挂与宿主系统隔离铁律
├── engine/              ← 可运行脚本（均"被调用"，不自动跑）
│   ├── load_soul.py        上下文拼装（被 mcp_server 的 recall_context 调用）
│   ├── record.py           主动记录接口（被 mcp_server 的 record_fact 调用）
│   ├── analyze.py          心理分析轻量辅助（可选，宿主 AI 也可自行推理）
│   ├── make_context.py     一键生成上下文（粘贴法用，用户手动跑）

│   └── README.md           三种集成方式技术细节

│   └── facts/              facts/prefs/bounds/commits.jsonl

```

> 一个文件夹 = 一个用户的记忆实例。多个 AI 想共享同一份"懂你"，就都挂同一个仓库文件夹；想隔离，复制文件夹即可。

## 2. 出厂行为契约（AI 必须做到的）

> 本契约约束的是**承载框架的 AI**，而非框架自身。框架是规则与机制，**不会自己执行任何动作**——是 AI 按本契约在每轮对话中行动（详见文末"脚本运行方式"）。

承载它的 AI 在**每一轮对话**中默认执行：

- **加载**：对话开始前，调用 `engine/load_soul.py` 的 `context_pack()`，把人格定义 + 已积累的用户事实 + 近期交互摘要注入自身上下文。
- **观察**：对话中持续识别用户的**事实陈述、偏好信号、情绪基调、行为模式、关系事件**。
- **记录**：把值得长期保留的内容，经 `engine/record.py` 写回 `data/facts/`。不等特点结束，实时增量写。
- **分析**：从语气/措辞/行为推断用户当前心理与长期倾向，结果进内部记录（不当面贴标签）。**强分析由承载 AI 依据 `rules/psychology.md` + `rules/user-analysis.md` 自行推理完成**；`engine/analyze.py` 仅作可选轻量关键词辅助（非诊断），AI 可 import 调用，但框架**不把它封成 MCP 工具**，以免弱匹配被误当结论。
- **对齐**：回应时，让积累的理解自然影响语气与措辞——让用户感到"被记得、被懂"，而非机械引用。

## 3. 与"训练"的区别（给用户的说明话术）

| 维度 | 传统微调训练 | 本框架 |
|------|------------|--------|
| 门槛 | 需数据/算力/GPU | 零，装好即用 |
| 持久化 | 焊进权重，难改 | 外部文件，可看可改 |
| 失忆风险 | 换模型即丢 | 框架在，理解就在 |
| 透明度 | 黑箱 | 每条事实可读可审 |
| 成本 | 高（重训） | 低（写文件） |

**结论**：用户不想训练、不会训练，本框架让他"直接聊就行"——理解在聊天中自然沉淀。

## 附：脚本运行方式（关于"会不会自己跑"）

**结论：除 `mcp_server.py` 外，所有脚本都不会自动运行。** 框架没有后台定时任务。

| 脚本 | 自动运行？ | 由谁触发 |
|------|-----------|----------|
| `mcp_server.py` | **是（被客户端拉起）** | MCP 客户端（WorkBuddy / Claude / Cursor 等）常驻管理，活着等活 |
| `engine/load_soul.py` | 否 | 被 `mcp_server` 的 `recall_context` 调用，或被开发者代码 `import` |
| `engine/record.py` | 否 | 被 `mcp_server` 的 `record_fact` 调用，或被开发者代码 `import` |
| `engine/analyze.py` | 否 | 可选辅助库，承载 AI 在对话中按需 `import` 调用 |
| `engine/make_context.py` | 否 | 用户手动跑，生成粘贴文本 |
| `engine/verify_soul.py` | 否 | 开发者/用户手动跑，检查结构完整性 |
| `examples/minimal.py` | 否 | 演示用，跑一次看效果 |

> 所谓的"自检脚本"（`verify_soul.py`）就是一份 18 项清单比对：逐个 `os.path.exists` 检查应存在的文件、校验 `config.schema.json` 合法 JSON、确保 `data/facts/` 四个空文件存在。纯静态检查，**不读你的记忆内容，不写入任何业务数据**（最多补一个空文件）。
>
> 若用户/宿主 AI 想要"定期自检记忆健康度（stale 标记、去重、增长管理）"，需由承载 AI 在轻量自检中主动触发，或用户手动跑——**框架本身不替你巡逻**。
